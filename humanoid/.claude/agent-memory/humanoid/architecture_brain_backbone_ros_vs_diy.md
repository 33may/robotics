**Settled stance (Anton, 2026-07-09):** the Oli brain is a **long-horizon, build-it-right system**, not a throwaway PoC. World-invariance + the `limxsdk` 3.8-ABI comms boundary are **settled/covered** — do NOT re-litigate them. The backbone question is **SETTLED by ADR-0002** (`docs/architecture/adr/0002-inner-dataflow-bus.md`, accepted 2026-07-06): inner comms is a **brokerless DIY dataflow bus** (BrainProtocols, architecture.md §7); **ROS2-as-backbone was explicitly considered and REJECTED** there (Fedora/rclpy tax; LimX's MROS is a closed wheel-gated fork so matching it buys nothing). Do NOT re-open this — the 2026-07-09 deliberation re-derived the same answer.

**What's actually still open:** (a) *which* DIY bus implementation — dora-rs vs Zenoh+MCAP vs custom (architecture.md §11, unpicked); (b) whether to pull ROS in as a **bridged nav-island** for reuse (Nav2/cuVSLAM). On (b), Anton chose **no-ROS (path A) for the SLAM PoC** on 2026-07-09. Current *code* reality: the bus isn't built yet — modules talk via **direct method seams** (`Teleop.to_policy_in → PolicyIn`, `GlideAction.step → GlideCmd`), so new reason modules are plain classes with clean contracts a future bus will carry, NOT bus nodes today.

**Analysis captured (so future sessions don't re-derive it):**
- ROS's value is NOT the bus (a socket is a weekend). It's the **ecosystem: drivers + classical-robotics algorithms + standard interfaces + introspection/record-replay tooling + battle-tested edge cases** (tf time-interp, QoS, clock skew). The bus is cheap; that stack is person-years.
- **ROS is radically extensible INSIDE a node** — any repo/language/tool wrapped in a thin `rclpy` shim. You do NOT lose "adapt any tool" by using ROS. It constrains only **inter-node contracts** (but custom `.msg` is allowed); real limits = the pub/sub + service + action interaction model, DDS operational model, IDL serialization. Not a straitjacket for a cognitive stack — its 3 primitives map cleanly onto voice/perception/memory/nav.
- **Key decomposition:** ROS reuse is **strong for classical robotics** (nav, SLAM, manipulation, sensor drivers) and **weak for ML-cognition** (memory, voice, detection, reasoning = PyTorch/HF/LLM, which get wrapped INTO ROS, not sourced from it). So DIY loses little on the ML modules, re-pays real cost on the robotics modules.
- **"We can always recreate ROS features":** true for any ONE feature; the trap is the aggregate + perpetual maintenance = becoming a middleware shop as a side effect of building a humanoid. Efficient allocation = **recreate the small subset you want perfect (pub/sub, message schema, record/replay, viz), borrow the expensive big-ticket items (Nav2, cuVSLAM, MoveIt) as bridged islands.**

**Leaning (mine, for Anton to rule on):** **DIY cognitive backbone + ROS as bridged islands** wherever its ecosystem pays off (nav/SLAM first). One bridge seam; backbone stays clean/ownable; adaptability AND reuse. Generalizes the "separable decision" from the nav path-A-vs-B talk. See [[nurec-gs-isaac-render]] / [[reconstruct-pipeline-milo]] for the SLAM-PoC context this arose in.

**Reversibility (why starting no-ROS is a two-way door):** migrating DIY→ROS later is cheap **iff** we pay a small insurance tax now, because the expensive part — the algorithms (cuVSLAM/PyCuVSLAM, planner, perception) — is transport-agnostic and **does not change** on migration; only the transport shim + message mapping + launch change. Insurance tax (≈<1 day, good architecture regardless): (1) **stamped, ROS-shaped messages** — every contract carries `frame_id`+`timestamp`, SI units, SE(3) poses → future `→ tf`/msg mapping is 1:1 not a translation; (2) modules talk **only through the bus**, never each other's internals; (3) a thin **`Bus` transport abstraction** so AF_UNIX is a swappable *backend* (ports-and-adapters) — the day we want ROS we add an `rclpy` backend and modules never know. Payoff: migration is **incremental, island-by-island** (Nav subsystem first), never a big-bang rewrite. Caveats: DDS-QoS/C++/multi-machine are *additive* new features not rework; the 3.8-`limx` wall is the same whenever hit; **`tf` retrofit is the one painful migration if stamping was skipped** — which is exactly why insurance #1 matters.

**How to apply:** Default the brain's own modules to the DIY backbone; reach for ROS only as an island with a clear ecosystem payoff, never as the mandatory backbone. Bank the reversibility insurance (stamped contracts + swappable `Bus`) from the first module written. Re-examine if the brain's inter-module needs keep growing toward "we keep reinventing another ROS feature" — that's the signal DIY has stopped paying. When Anton rules, promote to an ADR in `docs/architecture/adr/`.
